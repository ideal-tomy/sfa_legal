# 士業向けミニSFA（相談〜受任の見える化） 要件定義

---

## 1. このドキュメントの目的

- 弁護士事務所からの「士業向けミニSFAの実際に動くアプリdemoを見たい」という依頼に対し、**そのまま実装に着手できる粒度**でプロトタイプ要件を固定する。
- 目的は本番運用機能を全部作ることではなく、**相談受付から受任判断までの流れが一画面群で伝わること**と、**小さく始めて育てられる現実感**を示すこと。
- 本デモはまず **弁護士事務所向け** に最適化し、将来的に司法書士・行政書士・税理士等へ横展開しやすい構造を前提とする。

---

## 2. デモの狙い

### 2.1 伝えるべき価値

- 紙・Excel・メール・口頭で散っている「相談案件の進み具合」が、ひと目で追える。
- 「次に誰が何をするか」「フォロー漏れがないか」「どこで止まっているか」が分かる。
- いきなり重い基幹システムではなく、**まずは最小構成で回せる** 現実的なスタート案として見せる。

### 2.2 デモで避けること

- 本番CRMのような大量機能の詰め込み。
- 営業色の強い「パイプライン管理」表現。
- 実在クライアントを想起させる生々しい個人情報や事件情報の表示。

### 2.3 公開デモとして追加で伝えること

- 既存サイト上の紹介用サンプルではなく、**URLで共有して実際に触れる体験デモ** として成立させる。
- 「導入すると何が便利になるか」を 1〜3 分で体感できるようにする。
- 外部連携やAIは今回の本実装対象ではないが、**将来追加できる前提** が自然に伝わるようにする。
- 月額費用は固定約束の見せ方にせず、**小さく始める入口プランのイメージ** として伝える。

## 3. 想定ユーザー

| ユーザー | 主な関心 | このデモで見せる価値 |
|------|------|------|
| 代表弁護士・パートナー | 相談案件の抜け漏れ、受任見込、進捗の俯瞰 | ダッシュボードで全体をすぐ把握できる |
| 事務局長・事務スタッフ | 次回連絡日、資料待ち、担当の引き継ぎ | ボードと一覧で次アクションを追える |
| 若手弁護士・担当者 | 自分の案件の優先順位、面談後のフォロー | 相談詳細と期限超過の可視化 |

---

## 4. 代表シナリオ

### 4.1 シナリオA: 朝いちで全体を確認したい

1. 代表弁護士がデモを開く
2. ダッシュボードで「アクティブ相談件数」「今週フォロー」「期限超過」「受任見込」を確認する
3. 期限超過の案件をクリックして詳細へ移動する
4. 相談ボード上で現在ステージと次アクションを確認し、フォロー漏れ防止のイメージを掴む

### 4.2 シナリオB: 初回相談後に案件を進めたい

1. 事務スタッフが新規相談を登録する
2. 案件種別・紹介元・担当者・次回連絡日を入力する
3. 相談ボードに新しいカードが追加される
4. 初回面談後にステージを「利益相反確認」または「見積・委任契約案」へ更新する
5. ダッシュボードの件数と今週のフォロー一覧が自動で変わる

### 4.3 シナリオC: 受任見込を確認したい

1. 受任見込の案件を一覧またはボードから選ぶ
2. 詳細パネルで「相談概要」「次アクション」「見積レンジメモ」「紹介元」「担当者」を確認する
3. 受任に進んだ場合はステージを更新し、受任済みとして扱えることを確認する

### 4.4 商談時の60秒デモ導線

1. ダッシュボードで全体件数と期限超過を見せる
2. 期限超過一覧から1件を開き、詳細パネルで次アクションを確認する
3. 相談ボードへ移り、案件ステージを1段階進める
4. 新規相談を1件追加し、一覧とボードへ即時反映されることを見せる
5. 「ここにLINE/フォーム/AI要約などを将来連携できる」ことを補足する

### 4.5 商談時の3分デモ導線

1. 既存サイトからデモへ遷移し、初見でも迷わないことを見せる
2. ダッシュボードで「今どこで止まっているか」が分かることを説明する
3. 相談ボードで相談受付から受任までの流れを確認する
4. 詳細パネルで次アクション、紹介元、利益相反確認ステータスを確認する
5. 新規相談を追加し、ブラウザ内で体験が完結することを示す
6. 最後に連携プレビューと導入レンジ説明を見せ、将来拡張の余地を伝える

## 5. プロトタイプのスコープ

### 5.1 今回のMVPに含める

- ダッシュボード
- 相談ボード（カンバン）
- 顧客・案件一覧
- 相談詳細パネル
- 新規相談の簡易登録
- ステージ更新
- 次アクション日・次アクション内容の更新
- 今週フォロー / 期限超過の抽出
- 将来連携プレビュー表示（LINE / Webフォーム / AI要約 / 顧客DB連携などのイメージ表示）
- デモ用概算レンジ・含む/含まない・オプション説明
- デモデータのブラウザ内保持（セッション内、または `localStorage`）
- デモ初期化導線
- 架空データ・デモモード・未実装機能注記の表示

### 5.2 今回は含めない

- 認証基盤の本実装
- 外部カレンダー・メールとの実連携
- LINE、問い合わせフォーム、AI要約、顧客DBとの実連携
- 実データベース永続化
- 複数ユーザーの同時編集
- 帳票出力
- AI要約・自動分類の本実装
- 本番用監査ログ

## 7. ステージ設計

### 7.1 デモで使うステージ

| ステージID | 表示名 | 意味 |
|------|------|------|
| `intake` | 相談受付 | 問い合わせ受領直後、初回確認前 |
| `first_meeting` | 初回面談 | 初回面談前後、論点整理中 |
| `conflict_check` | 利益相反・資料待ち | 利益相反確認、必要資料待ち |
| `proposal` | 見積・委任契約案 | 費用説明、委任契約案、提案調整 |
| `retainer_likely` | 受任見込 | 受任可能性が高く、最終確認中 |
| `retained` | 受任 | 受任済み |
| `closed_lost` | 見送り | 見送り・保留・失注相当 |

### 7.2 集計ルール

- `closed_lost` はアクティブ件数から除外する。
- `retained` はアクティブ件数に含めないが、受任実績カードで別集計する。
- 期限超過は `closed_lost` を除外し、`nextActionDate < demoToday` の案件とする。
- 今週フォローは `demoToday` から 7 日以内の `nextActionDate` を持つ案件とする。

## 9. 機能要件

### 9.1 ダッシュボード

- KPIカードを4件以上表示する。
- 必須KPIは以下とする。
  - アクティブ相談
  - 今週のフォロー
  - 期限超過
  - 受任見込または受任済み
- 今週フォロー一覧から案件を選択すると、相談ボードの該当案件へ遷移できる。
- 期限超過一覧から案件を選択すると、詳細パネルが開く。

### 9.2 相談ボード

- ステージ列ごとに案件カードを表示する。
- 案件カードには最低限、件名、顧客名、次回アクション日を表示する。
- 案件選択時に詳細パネルを表示する。
- ステージ変更を1アクションで行える。
- ステージ変更後、ダッシュボードと一覧表示に反映される。

### 9.3 顧客・案件一覧

- 表形式で案件を一覧表示する。
- 少なくとも `clientName`、`contactName`、`referrer`、`practiceArea`、`lastContactAt`、`stage` を表示する。
- 行選択で相談ボードまたは詳細パネルへ遷移できる。

### 9.4 相談詳細パネル

- 件名、顧客名、相談概要、次アクション、次回アクション日、紹介元、担当者、現在ステージを表示する。
- 詳細内からステージ更新ができる。
- 詳細内から次アクション内容と日付を編集できる。

### 9.5 新規相談登録

- ダイアログまたはサイドパネルで実装する。
- 保存後、案件一覧と相談ボードの両方に即時反映する。
- 初期ステージは `intake` とする。
- 必須項目未入力時は保存できない。

### 9.6 概算レンジ説明

- 操作デモの直下に配置する。
- MVPの範囲、含む項目、オプション例、免責文を表示する。
- 表示文言は営業トークではなく、「この範囲ならこう始められる」という確認トーンにする。

### 9.7 連携プレビュー表示

- LINE相談取込、問い合わせフォーム起票、AI相談要約、顧客DB連携などを **将来拡張候補** として表示できる。
- これらは「対応可能」「連携イメージ」「オプション想定」などの文言で見せ、**実装済みと誤認させない**。
- 連携プレビューは固定文言またはダミーステータス表示とし、実際のAPI通信は行わない。
- 詳細パネルまたは下部説明セクションのいずれかで、自然に読める位置へ配置する。

### 9.8 公開デモ導線

- 既存サイトから1クリックでデモ画面へ遷移できる前提で構成する。
- 初見ユーザーでも何を試せばよいか分かる短い案内文を表示する。
- 画面内に「初期状態に戻す」導線を用意し、商談のたびに同じ状態から始められるようにする。
- デモ操作後に問い合わせまたは相談導線へ戻れることが望ましい。

### 9.9 価格表現の扱い

- 「月額1万円以内」は固定料金の約束として断定しない。
- 表現する場合は、軽量導入プランや小さく始める入口レンジとして扱う。
- 月額費用は運用範囲、サポート範囲、連携有無によって変動する前提を保つ。
- 利益源は運用支援、導入支援、マーケティング・ITコンサル、追加開発などの周辺提案を含み得るが、今回のデモ画面上では過剰に前面へ出さない。

## 10. 受け入れ条件

### 10.1 デモ成立条件

1. 初見ユーザーが 60 秒以内に「全体把握」「案件進捗確認」「新規相談追加」の3点を操作で理解できる。
2. ステージ変更または新規相談追加を行うと、ダッシュボードの件数と対象一覧が視覚的に更新される。
3. 期限超過案件が明確に区別され、フォロー漏れ防止の価値が伝わる。
4. 顧客一覧、相談ボード、詳細パネルの情報が矛盾しない。
5. モバイルでも主要操作（タブ切替、案件選択、詳細確認、ステージ更新）が成立する。

### 10.2 実装受け入れ条件

1. 画面は `dashboard` / `board` / `contacts` / `new consultation` の責務が分離されている。
2. デモデータはブラウザ内で更新可能で、画面再描画で失われない。
3. ステージ定義、集計関数、初期データは別ファイル化され、UI本体に埋め込まない。
4. `prefers-reduced-motion` 時も操作不能にならない。
5. 免責文に「架空データ」「正式見積ではない」を明記する。
6. 連携プレビューが、未実装であることを分かる文言つきで表示される。
7. デモリセット後に、定義済み初期データと導線文言が復元される。

- 活動履歴タイムライン

- Outlook / Google Calendar 連携
- 問い合わせフォームからの自動起票
- LINE相談窓口との自動起票
- AIによる相談要約・タグ付け
- 顧客管理DBとの同期
- 利益相反確認フローの厳密化

## 12. 実装フェーズ案

### Phase 1: デモの骨格

- ステージ定義を再設計する
- 既存3タブUIを新ステージ・新項目に合わせる
- ダッシュボード集計を更新する
- 顧客一覧にステージと担当者を追加する
- 初見向けの案内文とデモモード表示を置く

### Phase 2: 動く価値の追加

- 新規相談登録フォームを追加する
- 詳細パネルで次アクションを編集可能にする
- `localStorage` でデモ状態を保持する
- デモ初期化導線を追加する

### Phase 3: 魅力づけ

- 空状態、成功トースト、軽い導線テキストを追加する
- 将来連携プレビューを追加する
- 概算レンジ説明を現在のMVP要件に合わせて更新する
- モバイル詳細UIを磨く
- 既存サイトからの遷移導線と戻り導線を整える

- ダッシュボード、相談ボード、顧客・案件一覧、相談詳細、新規相談登録を実装する。

- デモ下部の概算レンジ説明を、今回のMVP範囲に整合する文言へ更新する。
